Executive summary

A user-supplied identifier passed to the order-lookup API was concatenated directly into a backend SQL query without parameterisation. An authenticated low-privilege user could inject a UNION-based payload to read arbitrary tables, including credentials and customer records, across the shared database. Because the database account was shared across tenants, the impact extended beyond the attacker's own data (scope change).

Affected asset

# endpoint (redacted)
GET https://api.xxxxxxxxxx.com/v2/orders/lookup?id={injectable}
# backend: MySQL 8.x · ORM bypassed via raw query builder

Description & root cause

The id query parameter was interpolated into a raw SQL string on the server side rather than bound as a parameter. Input validation only enforced a maximum length, which was insufficient to prevent injection.

The vulnerable code path bypassed the application's ORM in favour of a hand-built query for a "performance-optimised" reporting feature — a common root cause where a single un-parameterised query undermines otherwise safe data access.

Steps to reproduce

Sanitised

Target identifiers, parameters and payloads below are redacted or replaced with synthetic placeholders. Steps are summarised at the level published in a coordinated-disclosure write-up.

  1. Authenticate as a standard, low-privilege application user.
  2. Issue a request to the order-lookup endpoint and observe that a single quote in the id parameter triggers a database error reflected in the response (error-based oracle).
  3. Determine the column count via an ORDER BY probe, then confirm a reflected column position.
  4. Submit a UNION-based payload (illustrative, redacted target) to retrieve metadata:
-- illustrative payload (synthetic)
GET /v2/orders/lookup?id=0 UNION SELECT 1,@@version,database(),4-- -
-- response disclosed DB version + schema name

From there, standard enumeration of information_schema exposed table and column names, after which targeted reads were possible. Full extraction was not performed; a minimal proof sufficient to demonstrate impact was captured and reported.

Proof of concept

A single benign record was retrieved to evidence arbitrary read, with values redacted in this write-up:

Responsible handling

Testing stopped at minimal proof of read access. No bulk data was exfiltrated, modified, or stored. The PoC and a private payload log were shared with the triage team only.

Impact

  • Confidentiality: read access to arbitrary tables in a multi-tenant database, including hashed credentials and customer PII.
  • Integrity: the same injection point permitted stacked/derived write primitives under the shared DB account.
  • Scope: the shared database account meant one tenant could reach data belonging to others, escalating organisational impact.

Remediation

  • Parameterise all queries — use prepared statements / bound parameters for every dynamic value; never concatenate user input into SQL.
  • Re-enable the ORM for this path or wrap the reporting query in a vetted, parameterised builder.
  • Apply least-privilege database accounts scoped per tenant/role; remove the shared high-privilege account.
  • Add positive (allow-list) input validation for the id format (e.g. integer/UUID).
  • Deploy a WAF rule as defence-in-depth (not a primary control) and enable DB query logging/alerting.
