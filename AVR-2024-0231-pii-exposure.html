Executive summary

A public autocomplete/search endpoint intended to return only display names also serialised the full underlying user object, including email addresses, phone numbers, and partial date-of-birth. The endpoint required no authentication, exposing customer PII to anyone on the internet.

Affected asset

GET https://api.xxxxxxx.com/public/users/search?q=a
# no Authorization header required

Description & root cause

The API returned the raw user model instead of a trimmed view model. The frontend only rendered the name field, masking the over-exposure — but the sensitive fields were present in the JSON payload and retrievable directly.

This is a classic excessive data exposure combined with a missing authentication requirement on a sensitive resource.

Steps to reproduce

Sanitised

  3. Observe sensitive fields beyond the intended display name.

Proof of concept

Responsible handling

A single query returning a handful of records was captured purely to evidence field-level over-exposure. Records were redacted before reporting and not retained or processed further.

Impact

  • Mass harvesting of customer PII (email, phone, partial DOB) by an unauthenticated attacker.
  • Enables phishing, credential-stuffing target lists, and identity-fraud precursors.
  • Likely breach-notification and privacy-regulation implications.

Remediation

  • Return view models, not entities — serialise only the fields the consumer needs (here: display name + opaque id).
  • Require authentication and authorization for any endpoint touching personal data; reconsider whether public search is necessary at all.
  • Apply data-minimisation by default at the API layer.
  • Add automated tests asserting response schemas contain no sensitive fields.
