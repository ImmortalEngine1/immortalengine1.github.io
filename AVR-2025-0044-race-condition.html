Race condition in promo-code redemption allows repeated single-use redemption

Executive summary

The promo-code redemption flow checked remaining uses and then decremented them in two separate, non-atomic steps. By sending many redemption requests concurrently (TOCTOU), a single-use code could be redeemed multiple times before the counter updated, producing unlimited discount value.

Affected asset

POST https://shop.xxxxxx.com/api/v1/cart/redeem
# check-then-act on promo balance is not atomic / not locked

Description & root cause

The redemption logic performed a read ("is this code still valid?") and a later write ("decrement remaining uses") without a transaction, row lock, or atomic decrement. Concurrent requests all passed the validity check before any of them committed the decrement — a textbook time-of-check to time-of-use race.

Proof of concept

Responsible handling

Demonstrated with a low burst sufficient to prove multiple redemptions on a test cart; no real orders were completed and no funds moved.

Impact

  • Integrity / financial: single-use and limited-quantity codes can be redeemed repeatedly, enabling discount abuse and revenue loss at scale.
  • Generalises to other check-then-act flows (gift cards, referral credit, inventory holds).

Remediation

  • Make redemption atomic — use a single conditional/atomic decrement (e.g. UPDATE ... SET uses = uses - 1 WHERE uses > 0) and act on the affected-row count.
  • Wrap check-and-decrement in a transaction with appropriate row locking, or use an idempotency key per redemption.
  • Add server-side rate limiting and per-code concurrency guards.
  • Apply the same pattern review to all check-then-act business flows.
