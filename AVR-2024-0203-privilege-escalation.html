Vertical privilege escalation via mass assignment of role on profile update

Executive summary

The profile-update endpoint bound the entire request body to the user model. By adding a role attribute that the UI never exposes, a standard user could set their own role to administrator, gaining full access to privileged functionality including user management and data export.

Affected asset

PATCH https://app.xxxxxxx.com/api/v1/users/me
# body mass-assigned to the user entity without an allow-list

Description & root cause

The handler deserialised the JSON body and applied it wholesale to the user record (mass assignment / object injection). The role field was server-controlled by design but not protected from binding, so a client-supplied value overrode it.

Steps to reproduce

Sanitised

Target identifiers, parameters and payloads below are redacted or replaced with synthetic placeholders. Steps are summarised at the level published in a coordinated-disclosure write-up.

  1. Register / log in as a standard user.
  2. Capture the legitimate "update profile" request (e.g. changing display name).
  3. Add an unexpected privileged attribute to the JSON body and replay:
# synthetic body — note the extra field the UI never sends
PATCH /api/v1/users/me
{ "displayName": "test", "role": "admin" }

# response reflects elevated role
200 OK  { "role": "admin", ... }
  1. Re-authenticate / refresh session and confirm access to admin-only routes.

Proof of concept

Responsible handling

Elevation was demonstrated on a throwaway test account only. No real administrative actions were taken, no other users were affected, and the elevated account was reported for immediate de-provisioning.

Impact

  • Complete vertical privilege escalation from standard user to administrator.
  • Admin role granted user management, configuration, and bulk data export — i.e. full compromise of application confidentiality, integrity, and availability.

Remediation

  • Allow-list bindable fields — never bind request bodies directly to entities; use explicit DTOs that exclude server-controlled attributes like role.
  • Enforce server-side authorization for any role/permission change behind a separate, audited admin workflow.
  • Add integrity logging/alerting for role changes.
  • Regression test for mass-assignment across all write endpoints.
