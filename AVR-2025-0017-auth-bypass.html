Executive summary

The API accepted JSON Web Tokens without strictly pinning the signing algorithm. The verification library was configured to accept the algorithm declared in the token header, allowing a token to be presented that the server validated improperly — enabling forged tokens and authentication as arbitrary users.

Affected asset

Authorization: Bearer <JWT>
# verifier honoured the header "alg" instead of pinning a server-side algorithm

Description & root cause

The token verifier did not enforce an explicit, server-side allow-list of acceptable algorithms. When verification trusts the attacker-controlled alg header, the integrity guarantee of the signature collapses — a well-documented JWT pitfall (algorithm confusion / none-class issues).

Steps to reproduce

Sanitised

Target identifiers, parameters and payloads below are redacted or replaced with synthetic placeholders. Steps are summarised at the level published in a coordinated-disclosure write-up.

  1. Obtain a legitimate token by registering a normal account.
  2. Inspect the token header and observe the verifier honours the declared algorithm rather than a pinned one.
  3. Craft a token asserting a different subject/claims and present it (specific technique withheld from this public write-up).
  4. Observe the server accepts the manipulated token and authorises the request as the asserted user.

Responsible handling

Bypass was demonstrated only between two attacker-controlled test accounts. No real user account was accessed.

Impact

  • Full authentication bypass / account takeover of arbitrary users.
  • Attacker can read and modify victim data, defeating the application's primary trust boundary.

Remediation

  • Pin the signing algorithm server-side — explicitly allow-list one expected algorithm and reject everything else, including none.
  • Verify tokens with the correct key type for that algorithm; never let the token header choose the verification mode.
  • Rotate signing keys after the fix and shorten token lifetimes; add token-binding/claims validation (iss, aud, exp).
  • Add tests that reject tokens with unexpected alg values.
