IDOR on invoice endpoint exposes other tenants' invoices and PII

Executive summary

The invoice-retrieval endpoint authenticated the caller but failed to verify that the requested invoice belonged to the caller's account. Sequential, predictable invoice IDs allowed any authenticated user to enumerate and read other customers' invoices, which contained names, billing addresses, email addresses, and partial payment data.

Affected asset

GET https://app.xxxxxxxx.com/api/v1/invoices/{invoice_id}
# invoice_id: sequential integer, e.g. 100042, 100043 ...

Description & root cause

Object-level authorization was missing. The handler resolved the invoice by primary key and returned it after checking only that a valid session existed — never that the invoice's owner_id matched the authenticated principal.

Predictable sequential identifiers made enumeration trivial; a UUID would have raised the bar but is not a substitute for an authorization check.

Steps to reproduce

Sanitised

Target identifiers, parameters and payloads below are redacted or replaced with synthetic placeholders. Steps are summarised at the level published in a coordinated-disclosure write-up.

  1. Authenticate as User A and capture the request for one of your own invoices.
  2. Note the sequential numeric invoice_id.
  3. Decrement/increment the identifier by one and replay the request with User A's session.
  4. Observe a 200 OK returning an invoice owned by a different account.
# request (User A session) — synthetic id
GET /api/v1/invoices/10004X
Authorization: Bearer xxxxxxxxxxxxxxxx

# response: invoice belonging to a different tenant
200 OK  { "owner": "xxxxx", "email": "xxxx", ... }

Proof of concept

Responsible handling

Two adjacent IDs were accessed to demonstrate the access-control gap, then testing stopped. No mass enumeration was performed and retrieved third-party data was not retained.

Impact

  • Any authenticated user could read any other customer's invoices by walking the ID space.
  • Exposed data included PII (names, addresses, emails) and partial financial details — a privacy and regulatory exposure.
  • At scale, automated enumeration could harvest the entire invoice corpus.

Remediation

  • Enforce object-level authorization on every request: verify the resource's owner matches the authenticated principal (or an explicit grant exists).
  • Prefer non-sequential identifiers (UUIDv4) to reduce enumerability — as defence-in-depth, not the primary control.
  • Centralise access checks in middleware/policy layer so they cannot be forgotten per-handler.
  • Add rate limiting and anomaly detection for sequential object access patterns.
